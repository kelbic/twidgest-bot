# Twidgest Essayist — спецификация агента для автогенерации авторских разборов

**Версия:** 0.1 (initial draft)
**Дата:** 28.05.2026
**Статус:** проектирование, к реализации не начато
**Целевая интеграция:** дополнительный воркер в существующем `twidgest-bot` (или отдельный сервис)

---

## 1. Зачем

### Проблема, которую решаем

Существующий бот публикует короткие переводы виральных твитов и дайджесты. Это хороший формат для канала-новостника, но он **не позиционирует владельца как автора**. Канал выглядит как лента переводов, а не как медиа с редакторской позицией.

Для использования канала как полноценной витрины SaaS-продукта (целевая аудитория — медиа-агентства и блогеры) нужен **второй формат**: длинные авторские разборы в стиле vc.ru — со структурой, проверкой фактов, критическим разделом и выводами. Сейчас такие разборы пишутся вручную человеком в паре с LLM-ассистентом, занимая 30-60 минут на пост.

### Что хотим получить

Агент, который **раз в N часов** выбирает самый достойный твит из источников канала за период и публикует по нему длинный авторский разбор (3000-5000 знаков) — в Telegram-канал, с тем же качеством, что и ручной разбор.

### Чего НЕ хотим (важно)

- **Не AI-графомания.** Стандартный «напиши длинный пост по этому твиту» одним вызовом LLM даёт распознаваемый AI-текст без фактов. Это хуже текущего бота, а не лучше.
- **Не замена редактора.** Цель — генерировать черновики качества «человек бы опубликовал с минимальной правкой», а не «человек обязательно публикует автоматически» (опционально — режим human-in-the-loop).
- **Не быстрый прототип.** Серьёзный продукт требует серьёзной архитектуры — упрощения дают слабый результат.

## 2. Эталон качества — анализ существующих постов

Прежде чем проектировать, надо понять, **что делает разборы хорошими**. Это образцы, на которые равняемся. (См. файлы `vc_post_*.md`, выходные данные ручных сессий.)

### Структурный паттерн (8 успешных постов)

Все хорошие разборы соблюдают примерно такую структуру:

1. **Хук-абзац** — переворачивает ожидание читателя или ставит интригу. Никогда не «Сегодня поговорим о...».
2. **Что произошло** — голые факты без интерпретации, проверенные через первоисточник.
3. **Контекст / почему это важно** — что было до, чем отличается, что меняет.
4. **Деталь, которой нет в исходнике** — что-то, что бот сам бы не нашёл, требует поиска (как «Питер Тиль возглавил раунд» в посте про Halter, или «модель называется ER 1.6» в посте про Spot).
5. **Критический раздел / ограничения** — обязательно, иначе пост звучит как пресс-релиз. «Чего модель не умеет», «что пока остаётся неясным».
6. **Практический вывод или прогноз** — что с этим делать читателю.
7. **Подвал со ссылкой на канал** — короткий, мягкий.

### Что отличает хороший разбор от плохого

- **Точность цифр.** Несколько раз бот-новостник давал устаревшие или неверные числа («миллиард» вместо «$2 млрд» у Halter; «точнее МРТ» у TRIBE). Хороший разбор проверяет каждую цифру через первоисточник.
- **Живой ритм.** Длинные предложения чередуются с короткими. Машинная однородность ритма ловится читателем мгновенно (см. сам пост «Почему текст AI видно издалека»).
- **Свой угол.** Не пересказ новости, а **выбранный из новости тезис**. Эрдёш — «арка прогресса за 2 года». Halter — «бизнес-история, а не AI». Codex — «практики, переносимые в команду». Угол — это редакторская работа, и она сложнее всего.
- **Честность.** Если что-то нельзя проверить — это явно проговаривается («это не из официального анонса, это аналитика»). Если бот склеил два сюжета — разводим. Это и защищает от комментариев, и формирует доверие.

## 3. Архитектура — pipeline из 7 шагов

Pipeline спроектирован так, чтобы каждый шаг проверяем, заменяем и логируем. Не один LLM-вызов, а **agentic workflow** с разделением ответственности.

```
[1] Сбор кандидатов → [2] Селектор → [3] Поиск первоисточника →
[4] Фактчекинг → [5] Выбор угла → [6] Генерация черновика →
[7] Проверка стиля → [8] Публикация / очередь редактора
```

Если хоть один шаг отбраковал материал — pipeline останавливается, тема помечается как `not_publishable`, агент идёт к следующему кандидату.

### Шаг 1. Сбор кандидатов

**Вход:** канал-владелец (с источниками + параметрами).
**Выход:** список из 10-20 твитов за последние N часов (N конфигурируется, по умолчанию 12-24).

**Логика:**
- Использовать существующий `twitter_client` и кэш — не дублировать сбор.
- Фильтр по engagement: лайки выше X, retweets выше Y (берём пороги из настроек канала).
- Дедуп с уже опубликованными разборами (по `topic_signature` — переиспользуем существующий механизм из `topic_dedup`).

### Шаг 2. Селектор кандидата

**Цель:** из 10-20 кандидатов выбрать **один**, по которому стоит писать разбор.

Не все виральные твиты годятся для разбора. Хороший кандидат:
- Несёт **проверяемый факт** (анонс продукта, исследование, цифры). Личные мнения и эмоциональные реплики плохо разбираются — нечего фактчекать.
- Достаточно **значим** — не «компания X выпустила косметическое обновление», а событие с последствиями.
- Имеет **внешний контекст** для углубления (есть статья, пресс-релиз, paper).

**Реализация:** LLM-вызов (Sonnet/Opus) с промптом:
> Вот N твитов. По каждому оцени по шкале 1-10:
> (а) насколько твит несёт проверяемый факт, а не эмоцию;
> (б) насколько событие значимо;
> (в) насколько вероятно, что есть статья/paper/пресс-релиз с деталями.
> Верни JSON с оценками и кратким обоснованием выбора топ-1.

**Output:** ID выбранного твита + причина выбора (для логов).

### Шаг 3. Поиск первоисточника

**Цель:** найти **авторитетный источник** информации, на которой основан твит, чтобы не пересказывать сам твит, а опираться на оригинал.

**Реализация:** агентский веб-поиск.
- Извлечь из твита ключевые сущности (имена компаний, продуктов, людей, цифры) через LLM.
- Запросить веб-поиск (через какое-то API — нужно выбрать: Brave Search, Serper, или Anthropic web_search в API).
- LLM ранжирует результаты: предпочтительно официальный блог/anonce/paper > авторитетные издания (NYT, Reuters, IEEE, etc.) > вторичные пересказы.
- Выгрузить топ-3 страницы (через web fetch), извлечь текст.

**Output:** 1-3 источника с извлечённым текстом, маркированные как `primary` (официальный) / `secondary` (издание) / `tertiary` (агрегатор).

**Если ничего не нашли** — пометить материал как `unverifiable`, отбраковать.

### Шаг 4. Фактчекинг

**Цель:** проверить **каждое утверждение** твита на соответствие найденным источникам. Это критический шаг, ради которого вся затея.

**Реализация:** LLM (Sonnet/Opus) с промптом:
> Вот твит: [текст].
> Вот источники: [тексты топ-3 страниц с маркировкой].
> Для каждого фактического утверждения в твите верни JSON:
> - claim: точная цитата утверждения
> - verified: true/false/partial
> - source: ID источника подтверждения (или null)
> - correction: если utverdение неверно/устарело — каким должно быть верное

**Особенно важно ловить:**
- Устаревшие цифры (валуации, размеры команд, версии моделей).
- Преувеличения («миллиард» когда $2 млрд, «точнее X» когда «точнее предыдущих моделей X»).
- Склеенные сюжеты (твит про A, в подписи добавили B без оснований).

**Output:** JSON со списком проверенных утверждений + список исправлений.

**Если >30% утверждений `verified=false`** — материал отбраковать (бот соврал слишком сильно, разбор будет легко разнести).

### Шаг 5. Выбор угла

**Цель:** определить **редакторский угол** — не пересказ новости, а тезис, под который собирается пост.

Это самый сложный шаг — то, что я лично делаю как редактор:
- Halter → «бизнес-история о Тиле, не AI-новость».
- Эрдёш → «арка прогресса за 2 года, а не отдельный результат».
- TRIBE → «развести модель и стратегию Meta, не склеивать».

**Реализация:** LLM-вызов с шаблонами углов.

Определить 5-7 шаблонов углов и просить модель выбрать подходящий:
- `business_story` — фокус на деньгах, инвесторах, рынке (хорошо для стартапов).
- `progress_arc` — связь с предыдущими результатами в области, динамика.
- `practical_breakdown` — разбор «как это устроено и что забрать себе».
- `myth_busting` — «вот что все думают, а вот как на самом деле».
- `industry_signal` — «что это значит для индустрии в целом».
- `opinion_thesis` — провокационный тезис с обоснованием.

**Output:** выбранный шаблон угла + конкретный тезис («Halter — это пример того, как deeptech возвращается в недиджитализированные отрасли»).

### Шаг 6. Генерация черновика

**Цель:** написать пост по структуре раздела 2.1 с выбранным углом, проверенными фактами и исправлениями.

**Реализация:** LLM (Opus или сильный Sonnet) с большим system-промптом.

System-промпт должен содержать:
- Структурный шаблон (7 разделов из 2.1).
- Образцы хороших постов (2-3 примера в few-shot — например, Halter и TRIBE).
- Инструкции по стилю: «избегай однородности ритма, чередуй длинные и короткие предложения», «не используй вводные штампы вроде „важно отметить“».
- Запреты: «не упоминай факты, отсутствующие в данных от шага 4», «не используй цифры без указания источника», «не склеивай разные сюжеты».

User-промпт содержит:
- Угол и тезис от шага 5.
- Проверенные факты от шага 4 с источниками.
- Подвал-шаблон со ссылкой на канал.

### Шаг 7. Проверка стиля и фактов

**Цель:** автоматическая проверка черновика на типичные AI-проблемы перед публикацией.

**Реализация:** второй LLM-вызов с критическим промптом:
> Вот черновик поста: [текст].
> Вот проверенные факты: [JSON от шага 4].
> Проверь:
> 1. Все цифры в посте есть в проверенных фактах?
> 2. Нет ли утверждений, которых нет в источниках?
> 3. Есть ли критический раздел (ограничения, чего не хватает)?
> 4. Ритм текста — есть ли длинные и короткие предложения вперемешку?
> 5. Подвал со ссылкой на канал на месте?
> Верни JSON со списком проблем или `{"ok": true}`.

**Если есть критичные проблемы** — вернуть на доработку (один retry) или отбраковать.

### Шаг 8. Публикация — два режима

**Режим A: human-in-the-loop (по умолчанию для начала).**
- Готовый пост уходит в личный чат владельцу с кнопками «Опубликовать / Доработать / Отбросить».
- Это **обязательный режим на первые 2-4 недели** — чтобы накопить датасет «что человек принимает, что правит, что отбрасывает».
- Данные о решениях логируются — будут использоваться для тюнинга промптов.

**Режим B: автопубликация (включается опционально).**
- Только после того, как accept-rate в режиме A стабильно >80%.
- Даже в автопубликации — слать копию в личку владельцу для контроля.

## 4. Технические требования

### Стек
- Использовать существующую инфраструктуру: SQLAlchemy, aiogram, OpenRouter, twitter_client.
- LLM: для генерации — **Opus или GPT-4o-class** (не экономить, качество критично). Для классификации/проверки — Sonnet.
- Веб-поиск: выбрать провайдера (Brave Search API ~$5/1k запросов, Serper ~$5/1k, Anthropic web_search через API). Серверный, не из бота.

### Новые сущности БД
- `Essay` (id, channel_id, source_tweet_id, angle_template, status, draft_text, published_text, created_at, decided_at, decision)
- `EssayFactCheck` (essay_id, claim, verified, source_url, correction) — для аудита.
- `EssayConfig` (channel_id, enabled, frequency_hours, mode='hil'|'auto', threshold_engagement)

### Новые модули
- `agents/essayist/` — отдельная папка, не смешивать с существующими воркерами.
  - `candidate_selector.py` (шаг 2)
  - `source_finder.py` (шаг 3)
  - `fact_checker.py` (шаг 4)
  - `angle_picker.py` (шаг 5)
  - `drafter.py` (шаг 6)
  - `style_checker.py` (шаг 7)
  - `publisher.py` (шаг 8)
  - `pipeline.py` (оркестрация всех шагов)

### Воркер
- Отдельный scheduled job в `main.py`, запускается раз в N часов (конфигурируется, начать с 12).
- Параллельно для каждого канала с включённым `EssayConfig.enabled=True`.
- Конкретно для начала — **только для одного канала-витрины владельца**, не предлагать платящим пользователям.

### Логирование и наблюдаемость
- Каждый шаг pipeline пишет в БД: что подал на вход, что получил, сколько времени заняло, сколько потратил токенов.
- Метрики: accept-rate в HIL, среднее число fact-check ошибок на пост, средняя стоимость поста в токенах/долларах.
- В JOURNEY.md фиксировать каждый pivot стратегии.

### Стоимость одного поста — оценка
- Шаг 2 селектор: ~2k токенов Sonnet ≈ $0.01
- Шаг 3 поиск + fetch 3 страниц: ~$0.02 (поиск) + LLM на ранжирование $0.01 = $0.03
- Шаг 4 фактчекинг: ~15k токенов Sonnet (твит + 3 страницы) ≈ $0.05
- Шаг 5 угол: ~3k токенов ≈ $0.01
- Шаг 6 генерация Opus: ~8k токенов вход + 4k выход ≈ $0.30 (Opus дорогой)
- Шаг 7 проверка: ~6k токенов Sonnet ≈ $0.02
- **Итого: ~$0.40-0.50 за пост.** При 2 постах в день = ~$25/мес на один канал. При 100 постящих клиентах = $2500/мес расходов на LLM. Закладывать в pricing с самого начала.

## 5. Этапы реализации

Не делать всё сразу. **Фазы**, каждая со своей точкой выхода.

### Фаза 0: одноразовый прототип (1-2 дня)
- Один Python-скрипт без БД, без воркеров.
- Запускается руками: подаёшь твит, получаешь текст разбора.
- Минимальный pipeline: только шаги 3, 4, 6 (без селектора, без проверки стиля).
- **Цель:** проверить, что в принципе возможно получить разбор уровня «понравился редактору» через автоматику. Если результат хуже того, что я пишу вручную в чате с LLM — значит вся затея под вопросом.

### Фаза 1: MVP с HIL (3-5 дней)
- Полный pipeline (все 8 шагов).
- Минимальная БД для логирования.
- Telegram-кнопки «опубликовать/отбросить» для владельца.
- Запускается раз в день для одного канала владельца.
- **Цель:** накопить 30+ решений, измерить accept-rate.

### Фаза 2: тюнинг (2-3 дня)
- Анализ логов: какие посты отбраковываются, на каких шагах.
- Доработка промптов селектора и углов на основе реальных данных.
- Подключение автопубликации, если accept-rate >80%.

### Фаза 3: коммерциализация (отдельная история)
- Сделать как opt-in фичу для платящих пользователей.
- Отдельный тариф — `Essayist Pro` или подобный.
- Документация, onboarding, биллинг отдельного типа использования.

## 6. Открытые вопросы — решить до старта

1. **Какой выбрать провайдер веб-поиска?** Anthropic web_search через API (если ты на Claude), или внешний (Brave/Serper)? От этого зависит и интеграция, и цена.
2. **Где взять качественный fact-check датасет для оценки?** Без него непонятно, насколько хорошо работает шаг 4. Возможно, использовать наши 8 ручных постов как ground truth.
3. **Сколько вариантов одного поста генерировать?** Один или несколько с выбором лучшего? Влияет на цену.
4. **Как обновлять промпты по мере опыта?** Хардкод в коде, конфиг в БД, или внешний файл? Хорошо бы версионирование.
5. **Что делать с твитами без проверяемых источников?** Целые сегменты (опыт-посты, личные истории) не получится разбирать. Это норма или дыра в продукте?

---

## 7. Definition of done для Фазы 1

Считаем MVP готовым, когда:
- Pipeline стабильно работает раз в сутки на одном канале без падений в течение 7 дней.
- Из 7+ сгенерированных постов как минимум **3 опубликованы владельцем** без правок (accept-rate >40% как минимальная планка).
- Каждый шаг pipeline логируется в БД, можно поднять полный трейс любого поста.
- Нет ни одного поста, опубликованного с фактической ошибкой (= шаг 4 работает).

Если эти условия не выполняются — фаза 1 не закрыта, повторяем тюнинг.

---

## 8. Что мы НЕ делаем в этой версии

- **Изображения для постов.** Сейчас оставляем без картинок или используем картинку из source-твита. Генерацию обложек через DALL-E/Flux откладываем.
- **Перевод vc-постов в канал** (исходная идея). Это была другая задача, она решается тривиально и не нужна, если работает Essayist.
- **Поддержку нескольких языков.** Только русский на старте.
- **Веб-интерфейс для редактирования.** Только Telegram-кнопки accept/reject. CRUD через бота.
- **A/B тестирование разных промптов одновременно.** Сначала одна версия, потом улучшения.
